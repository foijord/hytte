# hytte — Kalsneset 27 in 3D

THREE.js reconstruction of the property Kalsneset 27, 4641 Søgne (gnr/bnr 437/109, Kristiansand), built from Kartverket 0.25 m LiDAR and a 16.5 cm Norge i bilder orthophoto. See `data/README.md` for data provenance and coordinate conventions.

## Run

```
python tools/serve.py
```

then open http://localhost:8123/. The server hosts the static files and the save API; three.js is vendored in `js/vendor/`, so no internet is needed.

## Layout

- `index.html`, `js/main.js` — the viewer: terrain mesh from the DTM heightmap with the orthophoto draped, sea plane at −0.45 m (DTM sea-surface pixels are sunk to −1.2 by make_heightmap.py so the plane reads as water), parcel boundary in yellow, and buildings as editable boxes (orange = on parcel 437/109, gray = neighbours).
- `web/heights.bin`, `web/meta.json` — Float32 heightmap generated from `data/dtm_25cm.tif` by `tools/make_heightmap.py` (default 0.5 m grid; run `python tools/make_heightmap.py 1` for full 0.25 m).
- `web/buildings.json` — parametric buildings generated by `tools/extract_buildings.py`. Simple footprints get one min-area rectangle; L/Z/T-shaped footprints (single-rect IoU < 0.82) are decomposed into up to 6 rectangles by greedy maximal-rectangle covering in the footprint's dominant-angle frame (part ids `<osmid>:<n>`). Base elevation comes from the DTM; the saddle-roof shape (ridge height, eave height, ridge axis, pitch) is least-squares-fitted to the LiDAR DOM inside each rectangle, competing against a flat-roof hypothesis on RMSE. Footprints trace the roof edge (Norwegian convention), so walls are inset by the overhang (default 0.4 m).
- `data/extra_structures.json` — structures not in OSM, currently the deck in front of the main cabin, generated by `tools/extract_deck.py` (which preserves manually added records): it flood-fills the LiDAR surface for a locally flat platform at cabin-floor elevation attached to the cabin footprint, and fits cabin-aligned rectangles. Merged into `web/buildings.json` on every `extract_buildings.py` run (type `deck`, drawn wood-brown).
- `data/` — source geodata (DTM, DOM, ortho, parcel polygon, OSM footprints).

Scene coordinates: meters, origin at the address point (E 71428.47, N 6458099.03 in EPSG:25833); x = east, y = height (NN2000), z = south.

## Editing buildings

Click a building to select it, then **t** = move, **r** = rotate (vertical axis only), **s** = scale, **d** = duplicate, **Del** = remove, **Esc** = deselect. Roof controls on the selection: drag the **yellow ridge handle** vertically to change the roof pitch, **g** toggles gabled/flat, **e**/**E** lowers/raises the eave line, **o**/**O** shrinks/grows the roof overhang. The HUD and the floating labels show live measurements (footprint, ridge and eave heights above the floor, pitch in degrees) while you adjust; labels toggle with the "labels" button. When a building is selected, the HUD shows **numeric fields** for w, d, eave and ridge — type a value and press Enter for exact dimensions.

**Saving:** **Ctrl+S** or the "save" button POSTs the scene to the dev server, which writes `web/buildings_edited.json`. The viewer prefers that file over the generated `web/buildings.json`, so re-running the extractors never clobbers your edits. "reset edits" deletes the edited file and reloads the generated baseline.

**Terrain cut:** the terrain is carved down to each box's base inside its footprint, with a 2.5 m smoothstep-graded falloff outward so the excavation blends into the slope; carved areas are tinted grey-brown so the blasting extent is visible (toggle with the "terrain cut" button). Recomputed automatically after every move/scale/delete.

**Terraform pads:** "add pad" drops a 6×6 m translucent green pad at the orbit point, at current ground level. Its base elevation levels the terrain to it — **cut and fill** — with the same graded falloff as excavations; move/rotate/resize it with the gizmos (Y arrow = target elevation) or the numeric fields, and delete it to undo. Pads save/load like any building and are walkable.

**Walk mode:** the "walk" button drops a 1.8 m person at the orbit point and takes their view (eye 1.7 m). WASD/arrows walk, drag looks around; on touch, the left third of the screen is a movement joystick and the rest looks. You can walk onto decks and slabs; Esc or the button exits, leaving the figure standing as a scale reference. `?walk=1` starts walking at `tgt`.

URL parameters for reproducible views: `?cam=x,y,z&tgt=x,y,z&labels=on&cut=off&new=A..E|off&sun=month,hour|off`. Defaults: new build variant A, sun simulation on (mid-June 15:00 solar), labels off.

**Sun study:** the "sun" HUD button enables real solar geometry for the property (58.06°N) with cast shadows; the two sliders set month and hour (solar time — local summer time runs ~1.5 h later). Use it to judge how the cabin shades the deck through the season.

## New build concept

The "new build" HUD button toggles between the existing buildings and the selected replacement design; the dropdown beside it picks the design from the list in `web/newbuild.json` (records tagged `variant`/`variantLabel`; shared records like the under-deck room show for every design). `?new=off` starts with the old layout, `?new=<key>` selects a design. The list currently holds four designs: **Saltdalshytta Frem 80** (default: BYA 80, BRA 67.5, 11.7 × 6.3 m, 27°, ridge → 8.06 NN2000), **Familiehytta Furutangen 75 med hems** (BRA 74 + ~38 m² hems, 11.15 × 7.6 m, 30°, ridge → 8.54), **Saltdalshytta Frem 95** (BYA 96, BRA 79.2, asymmetric 22° saddle with the ridge parallel to the 9.9 m sea facade, low plane over the veranda, ridge → 7.85, 3 bedrooms) and **Saltdalshytta Nova 120** (BYA 117, BRA 101, 14.7 m long side to the sea, 6.8° pulttak with the high window wall seaward → ~8.0; wall height estimated — not published). Each design is auto-fitted with: a grey concrete slab (0.35 m plinth) and the **deck/storage** — same width as the slab, connected to the sea wall, deck surface 6 cm below slab top, concrete storage room below (1.9 m headroom). There is no auto-leveling pad: natural rock meets the slab (minst mulig terrenginngrep); place manual terraform pads where targeted leveling is wanted. `tools/new_build.py` generates the records and the indicative floor plan in `docs/floorplan.svg` (final layout per manufacturer), deriving placement from the fitted existing-cabin records so regeneration keeps them aligned.
